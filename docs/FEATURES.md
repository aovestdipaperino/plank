# Plank-only features

plank tracks the `ds4_agent` C reference for the core agent loop, but moves faster on the user-facing side. This is the full list of things that exist only in plank; the README carries a short highlight reel and links here.

## Interface

- **Ratatui TUI** — the C reference is a plain line REPL; plank auto-selects a full-screen TUI when running on a terminal, with mouse-wheel scrollback.
- **Markdown renderer** — assistant replies render as styled markdown in the TUI via [ratatui-markdown](https://crates.io/crates/ratatui-markdown) (headings, emphasis, lists, tables), with tree-sitter syntax highlighting in fenced code blocks. The in-progress segment is re-rendered as tokens stream in, so partial emphasis and unclosed fences resolve live.
- **Animated status bar** — the spinner verb gets a Claude-Code-style shimmer sweeping across the word while the model works. The bar shows the working directory (home as `~`), the git branch after a powerline glyph, and context as a percentage, framed by a rule above and below the prompt; the throbber/verb/token-stats progress sits on its own line just below the output, so a turn stays visibly alive even when thinking is hidden.
- **Git-style diff cards** — an `edit` (or an overwriting `write`) renders a change card in the output: an `Update`/`Create(path)` header, an added/removed summary, and `@@` hunks with red-background removals and green-background additions (Myers diff via the `similar` crate). A `write` to a *new* file instead streams its content dimmed as it is generated, so you always see what lands on disk.
- **Word-wrapped prompt** — long input wraps onto the next row (word boundaries, hard-breaking overlong tokens) instead of scrolling sideways, keeping command/`!` highlighting intact across wraps.
- **Reports while busy** — `/context`, `/usage`, `/mcp`, and `/help` run mid-turn (answered from a turn-start snapshot) without interrupting the model; other commands still wait for the turn to finish.
- **Launch progress** — if the system-prompt KV cache must be rebuilt at startup, it warms behind a simple progress bar before the full UI appears; an up-to-date cache skips straight to the UI.
- **Type while it thinks** — each turn runs on a worker thread (the C reference's model-worker architecture), so the prompt stays live during generation: type the next message and press Enter to queue it. Queued messages join the conversation at the next tool round, or start the next turn when the current one settles. Esc interrupts; Ctrl-C clears the typed line first, then interrupts on an empty line.
- **Interruptible everywhere** — Esc stops generation *and* prompt prefill; the engine's cancel callback aborts mid-sync instead of making you wait out a long prompt.
- **Text selection and clipboard** — click-drag selects rendered output (WYSIWYG, wrapped lines included) and copies it to the system clipboard on release via `pbcopy` plus OSC 52 (so copy also works over SSH); Cmd-V pastes into the prompt through bracketed paste.
- **`@` file completion** — type `@` in the prompt to open a fuzzy typeahead over the repo's files, directories, and MCP resources, ranked with [nucleo](https://crates.io/crates/nucleo-matcher); Tab extends the common prefix, Enter drills into directories, and submodule paths are demoted so the project's own files come first. Honors `.gitignore` (tunable), quotes paths with spaces, and refreshes as files and MCP servers appear.

## Context and visibility

- **`/context`** — a visual breakdown of context-window usage by category (system prompt, tools, AGENTS.md, conversation).
- **`/usage`** — cumulative billed token usage for online models this session (input, output, and — on providers that cache prompts, like Anthropic — cache read/write with hit rate). Empty on the local engine, which has no billed usage.
- **Session-start context** — plank automatically injects git status, recent commits, discovered `AGENTS.md`/`CLAUDE.md` files, and the current date at the start of each session.
- **`/init`** — asks the model to analyze the codebase and generate an `AGENTS.md` for future sessions (build/test commands, architecture, gotchas).
- **Persistent memory (`/remember`)** — layered memory files loaded into session-start context: `~/.plank/MEMORY.md` (user scope) and `./.plank/MEMORY.md` (project scope). `/remember <text>` appends a dated note to the project file, `/remember user <text>` to the user file; the file template documents the four entry types worth keeping (user, feedback, project, reference). Oversized files are tail-truncated at injection so memory can't crowd out the conversation.
- **Task list (`task` tool, `/tasks`)** — the model keeps a structured, model-visible task list as working memory that **survives compaction**: a `task` tool (add/update/list) maintains it, the current list is injected into every turn, and it serializes with the session so it rides `/resume` and `/checkpoint` rollback. Displayed in three tiers: a `done/total` counter in the status bar, a contextual strip of the active plus next pending tasks above the input, and the full list via `/tasks`.
- **`/repro`** — dumps a diagnostic snapshot for bug reports to `~/.plank/repro/repro-<time>.md`: the exact rendered engine input (system prompt, tools, context, full transcript) plus the runtime knobs that shape generation (model, backend, context size, sampling, think mode, engine tuning). `/repro <note>` records a description of the bug. Read-only — the live session is untouched.

## Sessions and time travel

- **`/resume` and `/tag`** — `/resume` shows the most recent sessions numbered, with tag and last prompt; `/resume <number>` (or a sha prefix) continues one. `/tag <text>` labels the current session for the listings. Listing metadata is stored in a trailer record read with bounded head/tail reads, so `/list` and `/resume` never parse whole transcripts.
- **Session KV payloads and `/strip`** — `/save` also snapshots the engine's KV state to a fingerprinted sidecar (`<sha>.payload`) next to the transcript, so `/switch` and `/resume` restore it and skip re-prefilling the whole conversation. The fingerprint ties the payload to the exact model, system prompt, and transcript; anything stale is ignored and rebuilt by a normal prefill. `/strip <sha>` drops a session's payload to reclaim disk (the transcript survives), reporting the token count a later resume will re-prefill; `/list` shows each session's payload size or `stripped`.
- **`/checkpoint` and `/rollback`** — `/checkpoint <name>` snapshots the current conversation (transcript plus live engine KV) under a name; `/rollback <name>` truncates back to it and restores the KV, so a rollback resumes with no re-prefill. A rollback first auto-saves the current tail as `pre-rollback`, so it is itself undoable. `/checkpoint` with no argument lists the checkpoints. On an engine without KV snapshots (the echo stub) it degrades to transcript-only rollback.
- **`/clear`** — resets the session in place.

## Delegation and side questions

- **`/subagent`** — `/subagent <task>` delegates a task to a sidechain run of the same model: the subagent works with full tool access on a fork of the conversation, and only its final report is carried back into the transcript (framed as background context). The fork shares the conversation's KV-cache prefix going in and is rolled back afterwards, so delegation costs one report instead of the whole sidechain.
- **Named agents (`/agent`, `/subagent <name>`)** — agent definitions in `~/.plank/agents/*.md` (overlaid by `./.plank/agents`, frontmatter `name`/`description`, body = instructions) become dispatchable roles: `/agent` lists them and `/subagent <name> <task>` runs the sidechain with that definition's instructions. An unmatched first token falls back to the general-purpose subagent. (Parallel team orchestration is still tracked, pending per-session KV save/restore.)
- **`agent` tool (model-invoked delegation)** — the same sidechain machinery as `/subagent`, but callable by the **model**: it delegates a bounded `task` to a fresh scoped sub-agent (optional `name` selects a persona) and gets back only the final report, keeping the main transcript free of the intermediate research. Nesting is depth-capped so a sub-agent cannot itself delegate.
- **`/btw`** — ask a side question *while the agent is working* without disturbing the task. `/btw <question>` is answered from the shared conversation context with tools disabled and nothing written to the transcript (it never enters history, compaction, or the next turn). By default a mid-generation `/btw` genuinely **suspends** the running reply: the generation freezes, the aside is answered, and the task resumes byte-for-byte where it left off with zero re-prefill (`--disable-btw-suspend` falls back to the older pause-and-rerun boundary queue). While the answer streams the screen splits — main conversation 60% on the left, the side answer 40% on the right — and Esc cancels it and restores the full-width view. The side prompt reuses the live KV prefix, so it costs only the framed question.

## Tools and extensibility

- **`glob` tool** — the model can find files by name pattern across the tree (`**/*_test.rs`, `src/**/mod.rs`) instead of shelling out to `find`. `**` crosses directories, `*` stays within a path segment, results are relative and capped, VCS metadata is skipped.
- **Plan mode (`EnterPlanMode` / `ExitPlanMode`)** — a read-only propose-then-approve gate the model can enter itself: while active, `write`/`edit`/`bash` are refused and read-only tools (read/list/glob/search, web, MCP) stay available, so the model surveys the code before touching it. `ExitPlanMode` presents the proposed plan through the `ask` panel for approval (Approve lifts the gate, Keep-planning holds it); non-interactive runs auto-approve so scripts don't wedge.
- **`/skills` and the `skill` tool** — markdown prompt templates in `~/.plank/skills/<name>/SKILL.md` (overlaid by `./.plank/skills`) become slash commands: frontmatter gives `name`/`description`/`argument-hint`, the body is injected as the user turn with `$ARGUMENTS` substituted. The **model** can also invoke a skill itself via the `skill` tool (call with no name to enumerate), so a packaged procedure fires when the agent recognizes the task, not only when the user types the command; recursion is depth-capped.
- **Hierarchical MCP configs with `primaryTools`** — user-scope plus project-scope config files, and per-server control over which tool schemas go in the system prompt. `/mcp` inspects the state of connected servers.
- **MCP resources** — beyond tools, plank speaks the MCP *resource* protocol: `mcp_list_resources` and `mcp_read_resource` let the model reach content a server publishes as addressable resources (text inlines, binary reports type and size), and the same resources appear in `@` completion. Advertised only when a connected server actually publishes resources.
- **Expanded hooks** — command hooks from `~/.plank/hooks.json` + `./.plank/hooks.json` (reference-compatible shape). Beyond `PreToolUse`/`PostToolUse`/`Stop`, hooks fire on `UserPromptSubmit`, `SessionStart`, `SessionEnd`, `PreCompact`/`PostCompact`, and `PostToolUseFailure`; several can inject context into the turn. An optional JSON response envelope adds `continue:false`, non-blocking `systemMessage`, `suppressOutput`, and `async` execution over the exit-code protocol; matchers can key on arguments (`bash(git *)`, `write(*.md)`); and a `prompt` hook type feeds text to the model instead of running a process. `/hooks` lists what is configured.

- **`ask` tool** — when a turn is genuinely ambiguous, the model can pose a multiple-choice question (2–4 options, single or multi-select) instead of guessing or emitting a wall of prose. In the TUI it renders a selectable panel above the input (arrow keys move, Space toggles for multi, Enter accepts); the plain REPL prints a numbered list and takes a number or label prefix. Escape declines and the turn continues; in `--non-interactive` mode there is no user, so the tool returns immediately telling the model to proceed on its own rather than blocking.

## Shell and configuration

- **`!` commands** — `!<command>` runs a shell command immediately from the prompt (no model round-trip); output streams into the UI as it is produced, with exit-code display, and Esc/Ctrl-C kills it.
- **Mode-aware, directory-aware history** — arrow-key history on a `!` line cycles past shell commands only, and history is tagged with the directory it was entered in, so one project's commands stay out of another's (legacy untagged history stays universally visible).
- **Bash sandbox** — opt-in Seatbelt sandboxing for model-initiated shell commands (`--sandbox`, or `"enabled": true` in `~/.plank/sandbox.json` / `./.plank/sandbox.json`): commands run under `sandbox-exec` with writes limited to the working directory and temp dirs, plus configured `writablePaths`; `excludedCommands` glob patterns skip the sandbox (convenience, not a security boundary). Sandbox denials get a `[sandbox blocked: ...]` hint in the tool result so the model can react. User-typed `!` commands are never sandboxed — typing the command is the authorization.
- **`settings.json`** — persistent preferences for engine defaults, `@`-completion UI, safety defaults, the MCP timeout, and display toggles (`showToolCalls`, `showToolResults`, `showThinking`), layered user-scope then project-scope. A startup line names any setting in force, so a file that quietly selects the CPU backend can't hide.
- **`--ui-remote`** — a loopback control port for debugging convoluted TUI scenarios: `snapshot` captures the screen as ANSI text, `keypress` injects keys, and `uitree` returns the rendered layout as JSON, with replies ordered after the injected keys take effect so a harness never has to sleep.
